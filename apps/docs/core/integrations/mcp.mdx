title: MCP
description: Connect HyperFix to MCP clients like Cursor and Claude Desktop using the official HyperFix MCP server.

HyperFix supports the Model Context Protocol (MCP) in two ways:

1. **Built-in HTTP endpoint** — every HyperFix instance exposes `/api/mcp` out of the box. No extra setup, no extra process.
2. **Stdio package** (`@hyperfix/mcp`) — a local stdio server for clients that prefer stdio transport.

Both offer the same tools for managing workspaces, projects, tasks, comments, and labels.

## Built-in HTTP endpoint

Every HyperFix API instance ships with a Streamable HTTP MCP endpoint. This is the recommended approach for self-hosted deployments.

Point your MCP client at:

```
https://your-hyperfix-instance.com/api/mcp
```

On first connect, you will be redirected to HyperFix's login page. After you sign in, all MCP tools run as your authenticated user.

The endpoint implements OAuth 2.1 with PKCE for authentication. Discovery metadata is available at:

- `/.well-known/oauth-protected-resource/api/mcp`
- `/.well-known/oauth-authorization-server/api`

## Stdio package

The `@hyperfix/mcp` package runs a local stdio MCP server and authenticates with HyperFix using device authorization. Use this when your MCP client requires stdio transport.

### Prerequisites

- Node.js 20 or newer
- A running HyperFix API
- Access to the HyperFix web app to approve device login

By default, self-hosted HyperFix allows both `hyperfix-cli` and `hyperfix-mcp` as device auth client IDs.

### Install and register

Use the recommended interactive installer:

```bash
npx @hyperfix/mcp
```

The installer lets you choose where to register the server configuration:

- Cursor (user-wide)
- Cursor (project-level)
- Claude Desktop
- Custom JSON config path

You can also run the explicit install command:

```bash
hyperfix-mcp install --target cursor-user -y
```

Use `--help` to see all available targets and flags:

```bash
hyperfix-mcp install --help
```

### Point to a self-hosted instance

When generating config, pass your HyperFix base URL:

```bash
hyperfix-mcp install --target cursor-user -y --api-url https://hyperfix.example.com
```

Or set it via environment variable:

| Variable | Description |
| --- | --- |
| `KANEO_API_URL` | HyperFix API origin. Default: `http://localhost:1337`. Do not include `/api`. |
| `KANEO_MCP_CLIENT_ID` | Device auth client ID. Default: `hyperfix-mcp`. |

If you override `KANEO_MCP_CLIENT_ID`, make sure it is included in [`DEVICE_AUTH_CLIENT_IDS`](/core/installation/environment-variables).

### Authentication flow

On the first tool call that needs HyperFix access, the MCP server:

1. Requests a device code from HyperFix
2. Prints a verification URL and user code
3. Opens the browser when possible
4. Polls until you approve sign-in
5. Stores credentials in `~/.config/hyperfix-mcp/credentials.json`

## Available tools

Both the HTTP endpoint and stdio package expose the same tools:

- Session: `whoami`, `list_workspaces`
- Projects: `list_projects`, `get_project`, `create_project`, `update_project`
- Tasks: `list_tasks`, `get_task`, `create_task`, `update_task`, `move_task`, `update_task_status`
- Comments: `list_task_comments`, `create_task_comment`
- Labels: `list_workspace_labels`, `create_label`, `attach_label_to_task`, `detach_label_from_task`

## Debugging

To run the stdio server directly:

```bash
npx @hyperfix/mcp serve
```

When running from source in this monorepo:

```bash
pnpm --filter @hyperfix/mcp run build
pnpm --filter @hyperfix/mcp run start
```
